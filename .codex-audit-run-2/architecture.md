## Application and baseline

Nextcloud Desktop is a per-user Windows, macOS, and Linux synchronization client. It consists of a persistent Qt GUI/tray process, `nextcloudcmd`, classic sync roots, Windows CfAPI/VFS components, macOS FinderSync and File Provider extensions, and Linux/Windows file-manager integrations. It is primarily an outbound client: no public HTTP listener was found. Its network peers are configured Nextcloud servers over WebDAV/OCS plus optional push WebSockets; its inbound local surfaces are same-user sockets/named pipes, OS-vended XPC endpoints, command-line arguments, and `nc://` deep links.

The stack is C++20, Qt 6, CMake, SQLite journals, OpenSSL, QtKeychain, KF6Archive, KDSingleApplication, Swift 6.1/Apple FileProvider/NextcloudKit/Realm, Objective-C++, and Windows COM/CfAPI. Important entry points are `src/gui/main.cpp:52`, `src/gui/application.cpp:227`, `src/cmd/cmd.cpp:295`, `src/libsync/syncengine.cpp:487`, `src/gui/socketapi/socketapi.cpp:251`, `src/gui/macOS/findersyncxpc_mac.mm:35`, and `shell_integration/MacOSX/NextcloudFileProviderKit/Sources/NextcloudFileProviderKit/Extension/FileProviderExtension.swift:15`.

Comparable clients are ownCloud Desktop, Dropbox, OneDrive, Google Drive, and Syncthing. This class necessarily gives a configured server authority over remote account data and selected sync roots and gives the client broad local file access. A reportable bug must cross an intended file, account, process, origin, or privilege boundary. Server metadata, filenames, URLs, responses, and downloaded bytes remain untrusted even though the server is authoritative for account content. Same-user IPC without fresh account authentication is acceptable only when OS ACLs/entitlements establish the peer and commands stay inside the designed scope. Self-hosting and OEM support make compromised or malicious configured services relevant attackers.

## Trust model and controls

- The interactive OS user owns configured accounts and sync roots. The client usually runs without elevated privileges.
- The configured server is authoritative for authentication, WebDAV state, remote permissions, capabilities, E2EE certificates/metadata, push endpoints, and remote-wipe decisions.
- Account HTTP requests use the account SSL configuration (`src/libsync/account.cpp:448-503`). `AccessManager` enables HSTS and manual redirects (`src/libsync/accessmanager.cpp:27-43,75-87`). `AbstractNetworkJob` blocks HTTPS-to-HTTP redirects and strips authorization/credential injection on cross-host or cross-port redirects (`src/libsync/abstractnetworkjob.cpp:231-295`).
- App passwords become Basic authorization in `src/libsync/creds/httpcredentials.cpp:41-75`. Passwords, mTLS key material, and E2EE secrets normally use QtKeychain with insecure fallback disabled. Persistent cookies are serialized in the app config area (`src/libsync/cookiejar.cpp:90-139`).
- Non-macOS SocketApi uses a Unix runtime socket or Windows named pipe without an application-layer challenge (`src/gui/socketapi/socketapi.cpp:251-319,336-450`). Command-specific path/root/capability checks are the authorization boundary. A second Windows shell-extension server handles thumbnail/state JSON (`src/gui/shellextensionsserver.cpp:27-33,218-354`). macOS single-instance IPC uses `QLocalServer::UserAccessOption` (`src/gui/macOS/singleinstancemanager_mac.mm:19-55`).
- macOS FinderSync uses a registered Mach service and an intended same-team signing check before exposing an allowlisted adapter. File Provider and FPUI use OS-vended anonymous XPC endpoints; credentials are handed to the extension and stored in Keychain (`src/gui/macOS/fileproviderxpc_mac_utils.mm:157-295`, `.../FileProviderExtension.swift:638-744`).
- E2EE root metadata signatures are verified before metadata decryption, and metadata keys are checksum-checked (`src/libsync/foldermetadata.cpp:153-239`, `src/libsync/clientsideencryption.cpp:1478-1523`).
- Production GUI-testing IPC is compiled out by default (`CMakeLists.txt:240-245`). Dynamic PKCS#11/VFS loading and OEM endpoint overrides are privileged local/build configuration, not remote inputs by default.

## External inputs and dangerous sinks

1. WebDAV/OCS/XML/JSON: PROPFIND enumeration and property parsing (`src/libsync/networkjobs.cpp:70-90,181-425,810-915`), OCS/JSON jobs (`src/gui/ocsjob.cpp`, `src/libsync/networkjobs.cpp:1366-1470`), sync discovery (`src/libsync/discovery.cpp`, `src/libsync/discoveryphase.cpp`), lock responses, shares, activities, file actions, user status, and the current Assistant endpoints (`src/libsync/ocsassistantconnector.cpp`, `src/gui/tray/usermodel.cpp`). These inputs reach local paths, SQLite records, UI text/Markdown, browser-open calls, logs, and state-changing network requests.
2. File content and metadata: server-controlled names, hrefs, sizes, etags, checksums, permissions, direct-download URLs/cookies, E2EE metadata, live-photo metadata, and decompression. Important sinks include GET/PUT propagation (`src/libsync/propagatedownload.cpp`, `src/libsync/propagateupload.cpp`), rename/delete/mkdir jobs, temp files, `QDir`/`QFile`/`std::filesystem`, gzip/KArchive readers, and File Provider materialization/upload/move code.
3. Local IPC: textual SocketApi commands dispatch to `command_*` methods; Windows shell-extension JSON can trigger preview fetches and journal updates; Windows CfAPI hydration uses local servers; macOS FinderSync/File Provider/FPUI use XPC. Sinks include local deletion/move, VFS pinning, share/lock operations, browser/clipboard actions, remote preview requests, and credential transfer.
4. CLI/config/environment/deep links: GUI options are parsed in `src/gui/application.cpp:921-1029`; `nc://open` and `nc://login` are handled in `src/gui/urischemehandler.cpp:131-204`; `nextcloudcmd` parses target URLs, netrc, credentials, exclusions, and local/remote paths in `src/cmd/cmd.cpp` and `src/cmd/netrcparser.cpp`. QSettings, managed policies, theme/OEM configuration, proxy configuration, certificate bundles, environment variables, and security-scoped bookmarks feed network, filesystem, logging, dynamic-library, and process behavior.
5. Server-driven UI/integrations: activity links, call links, unified search URLs, private links, file-action URLs, and Assistant task/chat output reach `QDesktopServices`, `Qt.openUrlExternally`, QML `TextEdit` (including Markdown), clipboard, notifications, and logs. Process execution is concentrated in updater code, file-manager reveal helpers, and platform integration helpers.
6. Resource-sensitive parsers: many jobs call `reply()->readAll()` before JSON/XML/image processing; `QDomDocument`, `QXmlStreamReader`, JSON, gzip/KCompressionDevice, base64, image previews, and recursive enumeration/download are memory/CPU/disk exhaustion candidates. Downloads have decompression-safety and disk-space controls that must be checked on every alternate path.
